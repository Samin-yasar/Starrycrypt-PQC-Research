Overview

This release archives the complete implementation, benchmarking harness, and reproducible dataset accompanying research on browser-based post-quantum cryptography. It provides a permanent, citable snapshot of StarryCrypt-PQC, which implements FIPS 203 ML-KEM-768 with X25519 hybrid key exchange (per draft-ietf-tls-ecdhe-mlkem-04), targeting modern web browsers via both WebAssembly (WASM) and pure JavaScript execution paths.

What's New

Cryptographic Core

  • Complete ML-KEM-768 (FIPS 203) implementation, migrated from pre-finalization Kyber Round 3 to the NIST-standardized variant
  • Hybrid key exchange: ML-KEM-768 + X25519 with HKDF-SHA-256 shared-secret derivation
  • Strict FIPS 203 compliance: NIST-mandated hash domain separations (0x06, 0x1F) and the critical 33-byte dimension-parameter (k) concatenation

Benchmarking & Evaluation

  • Reproducible benchmarking harness addressing browser-specific measurement pitfalls (JIT warm-up instability, garbage collection interference, timer precision limitations)
  • 462-session benchmark dataset (N = 467 collected; n = 3 excluded for warm-up validation failure; n = 2 Chrome 87 outliers formally excluded) across 22 hardware configurations and 3 browser engines (WebKit, Blink, Gecko)
  • Controlled + organic composition: 307 lab sessions (BrowserStack, 67%) and 155 field sessions (Bangladesh and surrounding regions, 33%)
  • Time-based 200 ms warm-up phase (empirically validated: CV stabilizes below 5% after 150–180 ms across all engines) with automatic first-iteration outlier flagging

Performance Characterization

The primary finding is the browser-runtime vintage effect: Safari/WebKit and recent Chromium releases consistently outperform older Chromium and Gecko-based runtimes. Budget-tier devices running modern browser runtimes often outperform older flagship hardware running legacy runtimes, making runtime vintage the most robust predictor in this dataset.

Methodological Contributions

  • Browser measurement pitfall documentation: JIT warm-up instability, unreliable WebAssembly.validate() SIMD feature reporting, runtime-version confounding, and the difficulty of isolating cryptographic execution from browser-managed overhead
  • SIMD detection reliability finding: Seven Safari sessions reporting simd=false achieved sub-millisecond latencies (median 0.49 ms), falsifying the assumption that simd=false predicts poor performance. SIMD-based subgroup comparisons are therefore deprecated in this release.
  • Constant-time screening harness: Browser-native Welch's t-test heuristic for Fujisaki-Okamoto transform rejection screening (developmental-stage only; production requires formal TVLA validation)

Security Features

  • Constant-time Barrett reduction
  • Defense-in-depth memory zeroization via C volatile memset wrappers paired with JavaScript Uint8Array.fill(0) zeroing
Caveat: Zeroization is not guaranteed — JIT optimization, engine-internal buffer copies, and WASM linear memory visibility to same-origin JavaScript can defeat it. Unsuitable for long-lived secrets.

Known Limitations

  1. Uninstrumented overhead: 87% of measured handshake latency is browser-managed framework cost, not cryptographic core execution. Cross-implementation comparisons conflate framework integration with algorithmic efficiency.
  2. SIMD feature detection unreliability: WebAssembly.validate() self-reports do not reliably predict actual SIMD execution behavior. Do not use SIMD flags for performance stratification.
  3. Lab-dominant dataset: 67% of sessions are controlled lab runs. Field findings represent a Bangladesh-case study, not a globally representative sample.
  4. Zeroization limitations: Browser JIT optimization and runtime memory management may silently defeat explicit zeroization.
  5. Constant-time screening only: The Welch's t-test harness provides developmental screening, not production-grade side-channel resistance.
